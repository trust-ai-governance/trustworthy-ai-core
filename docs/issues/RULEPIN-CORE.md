# RULEPIN-CORE — 评测数字自带规则集指纹（Core 半边）

**Problem（plain language）：** 一个评测数字回答不了"**它测的是哪一版规则集**"。
今天 `provenance` 钉住了「跑的是哪批 WAL」（EV-PIN），却没钉「跑的是哪版规则」——
于是一个历史数字花了一整轮 git 考古才归因到具体的上游版本。

**Value：** 规则集指纹进 `provenance` 后，"同语料 + 规则从 A 版到 B 版 ⇒ 数字从 X 到 Y"
这句话**由 artifact 自证**，不再靠人翻 git。这是 measured>attested 的最后一块：
我们已经能证明"跑的是这批 WAL"，还要能证明"跑的是这版规则"。

> Dev brief. **前置**：无。**协作**：上游半边见 §4。**规模**：小。**不阻塞 PROV**（见 §2）。
> **注**：本 issue 的上游实现细节、具体版本号与归因案例在**私有对账文档**中，
> 本文只保留 Core 侧可独立实现所需的部分（见 [DISCLOSURE_POLICY](../DISCLOSURE_POLICY.md)）。

---

## 0. 现状（已核实，2026-07-19）

| 事实 | 状态 |
|---|---|
| 字段在 proto 里存在 | `DecisionTrace.policy_snapshot_version`（string） |
| 上游写入的值目前**非内容派生** | 已由上游确认（详见私有侧） |
| 🔴 **Core 侧未消费该字段** | `provenance` 里没有任何规则集信息 |

⇒ **即便上游改成内容派生，Core 不读它，pinned run 里照样不会出现规则集版本。**
所以这是**两侧各一半**的 issue，先做 Core 半边不亏（见 §4）。

### 0.1 为什么"翻 git 认定"不够

一次真实发生过的归因失败：某个数字对应的规则集变更，**其提交的 subject 完全没有提到它** ——
真正的规则集改动**搭便车藏在另一个 issue 的提交里**。定位最终靠内容特征词检索，
而不是靠提交信息或数字比对（后者还踩到过"同一分数在两版文档里含义相反"的坑）。

⇒ **提交信息不是可靠的版本记录。** 只有**记录自己携带的指纹**才是。

## 1. 范围（Core 侧做什么）

1. **读**：collect 扫描证据时，收集本次 run 观测到的
   `DecisionTrace.policy_snapshot_version` **取值集合**（是集合不是单值 —— 一个窗口内
   完全可能跨越一次规则集变更，那本身就是需要被看见的事实）。
2. **落盘**：写进 `provenance`，与 `window` / `wal_segments` **同级**：
   ```
   "ruleset": {
     "policy_snapshot_versions": ["<v>", ...],   // 观测到的全部取值，排序
     "distinct": 1,                              // >1 ⇒ 窗口跨越了规则集变更
     "source": "wal:DecisionTrace.policy_snapshot_version"
   }
   ```
3. **诚实的空值**：一条决策记录都没有 ⇒ `ruleset: null`（**不伪造**，同 EV-PIN §1.5-1 的纪律）。
4. **`distinct > 1` 要显形**：窗口跨越规则集变更时，该 run 的数字**不能归因到单一规则集版本**。

## 2. 🔴 关键约束：不得卡住 PROV 的近期闭环

PROV 要的是"某个数对应哪版规则集"，**用上游规则集文件的版本标识做印记就够** ——
语料侧本来就是这么钉的（git commit + 内容 sha）。

⇒ **PROV 先用版本标识闭环（已闭）；本 issue 独立推进，不是 PROV 的前置。**

**两者的区别值得写清楚**（这也是本 issue 为什么仍然必要）：

| | 事后认定的版本标识 | 内容哈希（本 issue 的目标） |
|---|---|---|
| 来源 | **人**事后翻 git 认定 | **记录自己**携带 |
| 可错性 | 会错（§0.1 就错过一次） | 结构上不会 |
| 第三方 | 要信我们的归因 | 自己比对哈希 |
| 成本 | 每次 O(一轮考古) | 一次性 |

## 3. 裁定与兼容

1. ✅ **`distinct > 1` 是否算 unpinned —— 已裁（2026-07-19）：
   artifact 层面不算；引用层面必须显形或收窄。**

   | 层面 | 规则 |
   |---|---|
   | **artifact** | **不算 unpinned** —— 窗口与 WAL 段仍是冻结的，数字仍可逐位复算 |
   | **对外引用** | 🔴 **必须收窄窗口，或明示"该窗口跨越规则集 A→B"** |

   **理由：** "把 X% 归因到规则集 A"而窗口实际跨了 A→B，**正是 §0.1 那次归因失败的形状**。
   **可复算 ≠ 可归因** —— `pinned` 保证的是**算得出同样的数**，
   它不保证**这个数只属于一版规则**。
2. **旧 bundle 的兼容**：本 issue 之前的 bundle 无此字段 ⇒ `ruleset: null`，
   与 `provenance: null` 同样按"honestly absent"处理，**不回填猜测值**。

## 4. 上游半边（不在本 issue，供对照）

上游把 `policy_snapshot_version` 改为**编译后规则集的内容哈希**，配置一改即变。
—— 在那之前，本 issue 记录到的仍是一个常量，**但管道是通的**：上游一改成哈希，
Core 侧无需再动，数字立刻自带指纹。**这也是先做 Core 半边不亏的原因。**

## 5. 验收

1. pinned run 的 `provenance.ruleset.policy_snapshot_versions` 非空且等于该窗口内实际观测值。
2. 无决策记录的 run ⇒ `ruleset: null`，**不是** `[]` 或伪造值。
3. 跨规则集变更的窗口 ⇒ `distinct > 1` 且在输出里显形。
4. 门禁不回归；`import treval` 不拉 web/PG。

> 🔴 **验收 3 的坑先写在这里**：当前环境里该字段的取值**恒为同一个常量**。所以任何拿真实 WAL
> 跑的测试都只会看到 `distinct == 1`，**测不出这条逻辑对不对**。守卫必须用**构造证据**：
> 造两种取值的证据、断言 `distinct == 2`。只断言 `distinct == 1` 的测试咬不住任何东西。
