# 公开仓披露纪律 —— 什么不能进这个仓，以及为什么

> **适用：** `trustworthy-ai-core` 是**公开仓**（GitHub `visibility: PUBLIC`）。
> **机械门：** `tools/check_doc_disclosure.py`，在 CI 里跑，命中 `error` 即 FAIL。
> **本文是那道门的说明书** —— 门告诉你"这条不行"，本文告诉你"为什么"和"改成什么"。

---

## 0. 为什么要有这道门

我们的工作天然跨私有/公开：引擎开源，运行时闭源，评测语料受限，选型涉及第三方厂商。
**靠人盯必然还会漏** —— 这已经是第 4 次同类问题（前三次：受限语料文本、内部数字↔配置
对照表、conformance fixture 产地）。前三次都被人接住了。

**第 4 次没接住：** 一段包含私有源码路径与控制缺口描述的文字**已经提交并推送到公开 main**
才被发现。发现它的是人工复核，而人工复核当时**恰好漏掉了另外两个文件**（一份 issue 文档
和一份 backlog），是这道门补扫出来的。

> **"每次都靠人接住" 不是控制，是运气 + 好记录。** 这是那个控制。

## 1. 五类不得公开的内容

| 类 | 严重度 | 为什么 |
|---|---|---|
| **control_gap** 自曝我方管控门实际没在跑 | 🔴 **最高** | 自己拆自己的台。一个把"可验证审计 / 防自证门"当护城河叙事的产品，公开宣布该门是空的，比任何外部批评都伤 |
| **vendor_named** 具名厂商 + 负面判断 | 🔴 高 | ① 触及《反不正当竞争法》§11 商业诋毁面；② 违反我方"不点名不贬损"纪律；③ 把采购谈判底牌亮给厂商本人 |
| **private_source** 私有运行时源码路径/行号 | 🔴 高 | 给出内部实现地图 |
| **platform_commit** 私有仓 commit 号 | 🔴 高 | 坐实"哪一次改动造成了什么" |
| **internal_name** 内部分层/SKU/私有文档名 | 🟠 中 | 泄露 roadmap 与包装姿态 |
| *private_doc_ref* 指向私有文档的**裸交叉引用** | 🟡 **warn，不 FAIL** | 只暴露"有这么份文档"，不含其内容。dev brief 里说"契约在上游某文档"是合法且必要的 |

### 1.1 为什么 `private_doc_ref` 只 warn

**一道会对合法交叉引用 FAIL 的门，一个月内必被关掉。** 而被关掉的门比没有门更糟 ——
大家以为它在跑。所以严重度分两级：**内容**必须拦（error），**指针**只需可见（warn）。
这条校准本身就是纪律的一部分，改门的人请先读这一节。

## 2. 中性化对照表（怎么改写）

| 私有原文 | 公开侧改写 |
|---|---|
| 具名厂商 | **候选 A / 候选 B**（不点名） |
| 私有源码路径 + 行号 | "上游侧已确认" |
| 私有仓 commit 号 | "上游规则集版本（号见私有对账）" |
| 内部分层名 / SKU 名 | "上游的规则层 / 语义层"；SKU 一律删 |
| "空门""实际没人跑""两端都空转" | "本门依赖的资产不在公开仓 ⇒ 归属见私有侧" |
| "某字段非内容派生" | "上游版本字段目前非内容派生（详见私有侧）" |
| **"未编译打包 ⇒ 泄露风险"** | **一律删** —— 这句本身就是最不该写的一句 |

## 3. 该留什么（别过度删）

**Core 必须能被独立实现。** 以下一律保留：

- harness 架构、`ProbeResult` / `VendorLabel` 契约、指标语义
- 方法学：双侧门、稳定性检查、质量/延迟分离、`corpus_sha` 冻结、fail-closed 判据
- 公开技术事实：开源软件行为、公开模型与其许可证、公开标准条款
- 实现必需的跨仓事实（"需上游提供 `content_class` 标签"）—— **去掉商业判断与内部架构名即可**
- **方法学层面的风险**（如"整条链同源会退化成自证"）——这是我们的方法论，是资产不是把柄

> 判据：**删掉的应该是"谁"和"哪里"，不是"怎么做"和"为什么"。**

## 4. 怎么用

```bash
python tools/check_doc_disclosure.py                 # 工作树（含未跟踪新文件）—— CI 跑这个
python tools/check_doc_disclosure.py --all-history    # 全历史：回答"这东西曾经发出去过吗"
```

**扫描范围包含未跟踪文件**，这是刻意的：历次事故都发生在**新文档**里，
而只读 `git ls-files` 的门会给新文件一路放行，直到 commit 那一刻 —— 那时已经太晚。

## 5. 命中之后

1. **别就地删** —— 内容通常是对的，只是**放错了仓**。先找它的私有归宿。
2. 现成的家：厂商评估 → 私有尽调文档 §7；上游实现细节 → 私有对账/设计文档。
3. 在公开侧留一行**指针 + 中性摘要**，让读者知道"这里有过判断，判断在私有侧"。
4. 若确实必须留在公开侧，按 §2 改写，**不要绕过门**（不要为了过门而把话说得含混到失真 ——
   那是另一种不诚实）。
